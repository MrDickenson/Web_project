# 1. Вибір архітектурного стилю: Моноліт

**Дата:** 03.12.2025
**Статус:** Прийнято

## Контекст
Ми розробляємо веб-застосунок для керування завданнями (To-Do System).
* **Стан проєкту:** Реалізовано початковий наскрізний функціонал.
    * Сервер надає один API-ендпоінт (`GET /items`).
    * База даних ініціалізована з таблицею `items` та тестовими даними.
    * Клієнтська частина налаштована на отримання та відображення списку цих записів.
* **Команда:** Один розробник.
* **Процеси:** Необхідність швидкого прототипування та налагодження базової бізнес-логіки без витрат часу на складну інфраструктуру.

## Рішення
Було вирішено використовувати **монолітну архітектуру** (Monolith).
Серверна частина (Backend), логіка роботи з базою даних та API для клієнта будуть реалізовані в межах одного проєкту/репозиторію.

## Альтернативи
* **Мікросервісна архітектура:** Була відхилена. Для To-Do системи, де сутності (Користувач, Завдання) тісно пов'язані, розбиття на мікросервіси створить невиправдану складність (network latency, distributed transactions) без реальної вигоди.
* **Serverless:** Відхилено на даному етапі через потенційну складність у локальному тестуванні та прив'язку до конкретного хмарного провайдера (vendor lock-in).

## Наслідки
* **Розробка:** Висока швидкість розробки. Вся логіка (наприклад, перевірка прав доступу користувача до конкретного завдання) знаходиться в одному місці.
* **Релізи:** Просте розгортання — достатньо запустити один екземпляр сервера та базу даних.
* **Масштабованість:** На старті достатньо вертикального масштабування. У разі зростання навантаження моноліт можна буде горизонтально масштабувати (запускати кілька екземплярів за балансувальником навантаження).
